## Contribution & Ownership Rules (recommended)

1. Small fixes (< 50 lines, docs, tests, low-risk refactor)
   - Owner: any contributor
   - Process: create PR, include unit tests if relevant, request one maintainers review.

2. Medium changes (new endpoints, new model usage, non-trivial UI flows)
   - Owner: Assigned module owner or ask owner to review
   - Process: create a PR, include tests, update README/architecture if behavior changes. Require 1-2 approvals.

3. High-risk changes (DB migrations, schema changes, model updates, infra)
   - Owner: module owner + ops
   - Process: discuss in issue/PR first, provide migration plan, include rollback/backup guidance, require 2 approvals.

4. Adding external dependencies
   - Always document why the dependency is required in the PR.
   - Small UI libs may be merged with one owner approval; backend/native extensions require owner + security review.

5. Secrets & credentials
   - Never commit secrets. Use environment variables and vault/secret manager for production secrets.

## Tests & CI

- The repo includes frontend unit & integration tests wired for `vitest` and `msw`. Keep tests fast and deterministic.
- Require test coverage for new backend logic and new critical flows that affect recommendations or payments.

---

## Oncall / Incident Response

- If recommendations stop working or LLM calls fail repeatedly:
  1. Check backend logs (structured logger) for errors in `app/logging` and `app/monitoring`.
  2. Verify `GROQ_API_KEY` and `NEON_DATABASE_URL` and any upstream network issues.
  3. If a model update caused regressions, roll back to previous `GROQ_MODEL` and open a postmortem.
